grpc-tc-mirror — runbook
========================

Run these blocks one by one. Copy the output of each diagnostic
step and send it back so the next step can be picked.


[0] Set the internal registry hostname (do this once per shell):

    export REGISTRY=<internal-registry>/main     # e.g. registry.example/main


[1] Diagnostic — what's inside build-tools (Debian + Python):

    sudo docker run --rm $REGISTRY/library/build-tools:latest bash -c '
        echo "=== os-release ===";
        cat /etc/os-release | head -4;
        echo "=== python ===";
        for p in python python3 python3.7 python3.8 python3.9 python3.10 python3.11; do
            command -v $p && $p --version 2>&1;
        done;
        echo "=== /opt ===";
        ls /opt 2>/dev/null
    '


[2] Diagnostic — TC base image apt feeds & external reach:

    sudo docker run --rm $REGISTRY/library/gcc84-build-x86_64:latest bash -c '
        echo "=== apt sources ===";
        cat /etc/apt/sources.list /etc/apt/sources.list.d/*.list 2>/dev/null \
            | grep -v ^# | grep -v ^$;
        echo "=== /opt ===";
        ls /opt 2>/dev/null;
        echo "=== curl github ===";
        curl -sI -m 10 https://github.com -o /dev/null \
            -w "github: %{http_code}\n" 2>&1 || echo "curl absent";
        echo "=== apt python search ===";
        apt-get update 2>&1 | tail -3;
        apt-cache search "^python3" 2>/dev/null | head -10
    '


[3] Build the tc-mirror image (run after the path is decided):

    cd <repo>/conan-recipes
    sudo docker build \
        --build-arg BASE_IMAGE=$REGISTRY/library/gcc84-build-x86_64:latest \
        -f Dockerfile.grpc-tc-mirror \
        -t grpc-tc-mirror .


[4] Run the build and collect artefacts (x86_64):

    mkdir -p output
    sudo docker run --rm \
        -v "$(pwd)/output:/work/conan-recipes/output" \
        grpc-tc-mirror

    ls -la output/*.nupkg


[5] Cross-build for arm or arm64:

    Profiles in profiles/ are tuned for the matching CI image.
    For cross-builds Conan needs separate host and build profiles:
      PROFILE       — target architecture (arm / arm64)
      PROFILE_BUILD — native architecture of the docker host (x86_64)

    The ARM base images in ProGet are tagged 0.1.0 (no :latest manifest).
    test_arm_cross.sh respects $BASE_IMAGE_TAG (default 0.1.0); use the
    same tag here when invoking docker build directly.

    arm (32-bit, hard-float, gcc 7.5 linaro):
        sudo docker build \
            --build-arg BASE_IMAGE=$REGISTRY/library/gcc75-build-arm:0.1.0 \
            -f Dockerfile.grpc-tc-mirror -t grpc-tc-mirror-arm .

        sudo docker run --rm \
            -e PROFILE=/work/conan-recipes/profiles/lin-gcc75-arm-linaro \
            -e PROFILE_BUILD=/work/conan-recipes/profiles/lin-gcc84-x86_64 \
            -v "$(pwd)/output-arm:/work/conan-recipes/output" \
            grpc-tc-mirror-arm

    arm64 (gcc 7.5 linaro):
        sudo docker build \
            --build-arg BASE_IMAGE=$REGISTRY/library/gcc75-build-arm64:0.1.0 \
            -f Dockerfile.grpc-tc-mirror -t grpc-tc-mirror-arm64 .

        sudo docker run --rm \
            -e PROFILE=/work/conan-recipes/profiles/lin-gcc-aarch64-linaro \
            -e PROFILE_BUILD=/work/conan-recipes/profiles/lin-gcc84-x86_64 \
            -v "$(pwd)/output-arm64:/work/conan-recipes/output" \
            grpc-tc-mirror-arm64

    Heads-up: the standalone Python tarball shipped under packages-linux/
    is x86_64-only. Cross-images run docker natively on x86_64, so the
    interpreter inside is still x86_64 — that is fine for running Conan;
    the cross-toolchain inside the image targets arm/arm64 binaries.


[6] Automated tests for the cross-build path:

    Driver script: test-astra/test_arm_cross.sh
    Modes:
      smoke  pre-flight + image probe + docker build  (~1-2 min)
      build  full Conan tree + verify 7 .nupkg names  (~15-25 min)

    Run smoke first to catch image / toolchain problems quickly:
        export REGISTRY=<internal-registry>/main
        ./test-astra/test_arm_cross.sh smoke arm
        ./test-astra/test_arm_cross.sh smoke arm64

    If smoke is green, run the full build:
        ./test-astra/test_arm_cross.sh build arm
        ./test-astra/test_arm_cross.sh build arm64

    Each block prints PASS/FAIL lines and exits non-zero on failure.
    Logs land in /tmp/{pull,probe,build,run}.log.

[8] Diagnostic — ARM build mode fails (after step 4):

    When `./test_arm_cross.sh build arm` fails inside the Conan tree
    (typically abseil or grpc), the visible terminal tail only shows
    a make caescade — the actual gcc error is higher up in /tmp/run.log.
    Run these three blocks one by one and share their output.

    [a] First real compiler error (not the make summary):
        grep -m 3 -B 1 -A 4 "error:" /tmp/run.log | head -50

    [b] How Conan interprets the host + build profiles
        (verifies that user_toolchain / buildenv / compiler_executables
        actually got picked up):
        sudo docker run --rm grpc-tc-mirror-arm bash -c '
            conan profile show \
                -pr:h /work/conan-recipes/profiles/lin-gcc75-arm-linaro \
                -pr:b /work/conan-recipes/profiles/lin-gcc84-x86_64
        '

    [c] Sanity-check the cross-gcc directly (catches a "GCC 7+ required"
        failure in seconds without waiting for the 15-min build):
        sudo docker run --rm grpc-tc-mirror-arm bash -c '
            /opt/linaro-arm-7.5.0/gcc-linaro-7.5.0-2019.12-rc1-x86_64_arm-linux-gnueabihf/bin/arm-linux-gnueabihf-gcc --version;
            /opt/linaro-arm-7.5.0/gcc-linaro-7.5.0-2019.12-rc1-x86_64_arm-linux-gnueabihf/bin/arm-linux-gnueabihf-gcc -dumpversion;
            echo "int main(){return 0;}" | /opt/linaro-arm-7.5.0/gcc-linaro-7.5.0-2019.12-rc1-x86_64_arm-linux-gnueabihf/bin/arm-linux-gnueabihf-gcc \
                --sysroot=/opt/linaro-arm-7.5.0/sysroot-glibc-linaro-2.25-2019.12-rc1-arm-linux-gnueabihf \
                -x c - -o /tmp/canary && file /tmp/canary
        '


    [d] Show what Conan actually writes into conan_toolchain.cmake
        (the only file CMake reads to pick CMAKE_C_COMPILER etc).
        Uses a minimal conanfile.txt — no third-party packages, no
        network. If CMAKE_C_COMPILER / CMAKE_CXX_COMPILER are missing
        in the output, neither [conf] tools.build:compiler_executables
        nor user_toolchain are propagating:
        sudo docker run --rm grpc-tc-mirror-arm bash -c '
            mkdir -p /tmp/probe && cd /tmp/probe;
            printf "[generators]\nCMakeToolchain\n" > conanfile.txt;
            conan install . \
                -pr:h /work/conan-recipes/profiles/lin-gcc75-arm-linaro \
                -pr:b /work/conan-recipes/profiles/lin-gcc84-x86_64 \
                --no-remote 2>&1 | tail -10;
            echo "=== conan_toolchain.cmake ===";
            cat /tmp/probe/conan_toolchain.cmake
        '


    [e] After a build run, check whether linaro-arm.cmake actually
        loaded for each package. The toolchain prints a unique marker
        "==LINARO-ARM-TC==" when CMake includes it. Count occurrences
        and look for which packages did NOT load it:
        grep -c "==LINARO-ARM-TC==" /tmp/run.log
        grep -B 1 "Building CXX" /tmp/run.log | grep -E "Building CXX|LINARO-ARM-TC" | head -40

        If the marker is missing right before the abseil build banner,
        Conan generated a conan_toolchain.cmake without our user_toolchain
        for that package — even though the host profile says it should.


    [f] Show what CMake actually wrote into CMakeCache.txt as the
        compiler (the only place make/ninja read it from). If the
        toolchain runs but build still fails, this is the deciding
        evidence.
        sudo docker run --rm grpc-tc-mirror-arm bash -lc "
            cd /work/conan-recipes;
            conan export abseil --version=20250127.0 2>&1 | tail -3;
            conan install --requires=abseil/20250127.0 \
                -pr:h /work/conan-recipes/profiles/lin-gcc75-arm-linaro \
                -pr:b /work/conan-recipes/profiles/lin-gcc84-x86_64 \
                --build=missing --no-remote \
                -s build_type=Release \
                -o '*/*:shared=True' 2>&1 | tail -15 || true;
            echo '=== CMakeCache.txt compiler vars ===';
            find /root/.conan2/p/b -path '*absei*' -name CMakeCache.txt \
                -exec grep -E 'CMAKE_C_COMPILER:|CMAKE_CXX_COMPILER:|CMAKE_AR:|CMAKE_SYSROOT:' {} +
        "


    [g] abseil-specific failure (int128.cc.o + cctz/time_zone_*.cc.o):
        This is the classic "host profile leaked x86 flags" or "cppstd/libcxx
        wrong" signature. Run these in order and stop on the first FAIL.

        1. The first real compiler error (not the make Error 1 cascade):
              grep -m 1 -B 2 -A 6 "error:" /tmp/run.log

        2. Confirm the host profile is the ARM one and has no x86 leftovers
           (no -m64, no -march=core2, libcxx=libstdc++11, cppstd=17):
              sudo docker run --rm grpc-tc-mirror-arm bash -c '
                  conan profile show \
                      -pr:h /work/conan-recipes/profiles/lin-gcc75-arm-linaro \
                      -pr:b /work/conan-recipes/profiles/lin-gcc84-x86_64
              ' | grep -E "arch=|compiler=|libcxx=|cppstd=|cflags|cxxflags"

        3. Sanity-check that the cross-gcc actually supports __int128
           (this is what int128.cc.o needs; if it does not, abseil falls
           back to a soft impl — but only if HAVE_INTRINSIC_INT128 is
           detected correctly):
              sudo docker run --rm grpc-tc-mirror-arm bash -c '
                  GCC=/opt/linaro-arm-7.5.0/gcc-linaro-7.5.0-2019.12-rc1-x86_64_arm-linux-gnueabihf/bin/arm-linux-gnueabihf-gcc;
                  SYSROOT=/opt/linaro-arm-7.5.0/sysroot-glibc-linaro-2.25-2019.12-rc1-arm-linux-gnueabihf;
                  echo "__SIZEOF_INT128__:";
                  echo "" | $GCC --sysroot=$SYSROOT -dM -E -x c - | grep INT128;
                  echo "C++17 chrono compile test:";
                  echo "#include <chrono>
                  int main(){ return std::chrono::seconds(1).count();}" \
                      | $GCC --sysroot=$SYSROOT -std=c++17 -x c++ - -o /tmp/chr 2>&1 | head -20
              '

        4. Replay just abseil with -j1 to get a clean, sequential error log
           (parallel make hides the first failure under noise):
              sudo docker run --rm grpc-tc-mirror-arm bash -lc "
                  conan create /work/conan-recipes/abseil --version=20250127.0 \
                      -pr:h /work/conan-recipes/profiles/lin-gcc75-arm-linaro \
                      -pr:b /work/conan-recipes/profiles/lin-gcc84-x86_64 \
                      --build=missing --no-remote \
                      -s build_type=Release \
                      -o '*/*:shared=True' \
                      -c tools.build:jobs=1 2>&1 | tee /tmp/abseil-arm.log | tail -40
              "
              # Then: grep -m 1 -B 2 -A 6 "error:" /tmp/abseil-arm.log

        Common verdicts:
        - "unrecognized command line option '-m64'" / "-march=core2" →
          host profile copied from x86_64 without scrubbing flags.
          Fix: edit profiles/lin-gcc75-arm-linaro, drop the x86 cflags.
        - "fatal error: bits/c++config.h" / "no such file" →
          sysroot path missing or wrong in user_toolchain. See [8d].
        - "'__int128' is not supported on this target" →
          armv7hf gcc 7.5 should have it; if missing, the toolchain is
          mis-detected. Re-run [c] in step [8] and verify -dumpversion
          prints 7.5.0, not the host gcc.
        - "C++17 not supported" / "<filesystem>: No such file" →
          host profile lacks `compiler.cppstd=17` or
          `compiler.libcxx=libstdc++11`.


[8h] Toolchain did not propagate to a transitive dep (after [8g] verdict #3):

    Symptom: `[8e]` count is < 7, and grep shows ==LINARO-ARM-TC== for
    some packages but not for the failing one (typically abseil, re2,
    or protobuf — pulled in transitively by grpc).

    The fix in commit e9bdb19 (`*:tools.cmake.cmaketoolchain:user_toolchain`)
    relies on Conan 2.x propagating the conf to every node of the dep
    graph. If a single node still misses the toolchain, conan_toolchain.cmake
    for that node will not include linaro-arm.cmake, and CMake falls back
    to the system compiler (Stretch g++ 6.3 → policy_checks.h trips with
    "GCC 7 or higher").

    Three checks (no rebuild, just inspect the failed build folder):

    1. Confirm the host profile has the `*:` prefix on user_toolchain:
           grep user_toolchain profiles/lin-gcc75-arm-linaro
       Expected: a single line starting with
       `*:tools.cmake.cmaketoolchain:user_toolchain=`. Without `*:`,
       only the consumer node receives it.

    2. Inspect conan_toolchain.cmake for the failing package — this
       is the only file CMake reads.

       CRITICAL gotcha: test_arm_cross.sh runs the build with
       `sudo docker run --rm`. The `--rm` flag destroys the container
       (and its /root/.conan2) the moment the build exits. So
       /root/.conan2/p/b/abseid* never appears on the host at all —
       even with sudo+bash -c the path is genuinely gone.

       To make the cache survive, mount it onto the host before
       re-running:

           mkdir -p "$(pwd)/conan-cache"
           sudo docker run --rm \
               -e PROFILE=/work/conan-recipes/profiles/lin-gcc75-arm-linaro \
               -e PROFILE_BUILD=/work/conan-recipes/profiles/lin-gcc84-x86_64 \
               -v "$(pwd)/conan-cache:/root/.conan2" \
               -v "$(pwd)/output-arm:/work/conan-recipes/output" \
               grpc-tc-mirror-arm \
               bash -lc './test-astra/run_test_grpc.sh || true'

       The trailing `|| true` lets the build fail without aborting
       the script; the bind-mounted cache survives. Then on host:

           # Capture the path from find (works regardless of the
           # actual hash — the abseil prefix may be abseidNNN, abseiZNNN,
           # absei2NNN, etc., depending on Conan settings):
           TC=$(sudo find "$(pwd)/conan-cache/p/b" \
                   -name 'conan_toolchain.cmake' -path '*absei*' \
                   2>/dev/null | head -1)
           echo "$TC"
           sudo cat "$TC" \
               | grep -E "user_toolchain|linaro|CMAKE_C_COMPILER|CMAKE_CXX_COMPILER|CMAKE_AR|CMAKE_SYSTEM" \
               | head -30

       Quick alternative without re-running (if you have a still-running
       failed container — drop --rm next time, or `docker run` it
       interactively): use `docker cp` from the live container:

           CID=$(sudo docker ps -a --filter ancestor=grpc-tc-mirror-arm --format '{{.ID}}' | head -1)
           sudo docker cp \
               "$CID":/root/.conan2/p/b/. /tmp/conan-b/
           sudo find /tmp/conan-b -name 'conan_toolchain.cmake' -path '*absei*'

       If conan_toolchain.cmake exists but does NOT contain
       `include(.../linaro-arm.cmake)`, the conf did not propagate
       for that package, regardless of what the profile says.

       Concrete signature confirmed 2026-05-07: the abseil
       conan_toolchain.cmake had ONLY the placeholder header

           ########### 'user_toolchain' block ##############
           # Include one or more CMake user toolchain from tools.cmake.cmaketoolchain:user_toolchain

       and nothing else (no include line, no CMAKE_C_COMPILER,
       no CMAKE_AR, no CMAKE_SYSTEM_*). That is the unambiguous
       fingerprint of "Conan rendered the template but the conf
       value was empty for this node". Go to step 3 below.

    3. If the toolchain is missing only for some packages, force it via
       -c on the CLI (bypasses any [conf] pattern-matching pitfalls).

       Three traps to avoid (learned the hard way 2026-05-07):

       a) `--build=missing` only rebuilds packages whose binary is
          missing. To guarantee a fresh CMake run for the failing
          package, ADD an explicit `--build=abseil*` (Conan accepts
          multiple --build patterns).
          DO NOT use `conan remove "abseil/*" -c` — that nukes the
          recipe too, and with --no-remote the version range
          `abseil/[>=20230802.1 <=20250127.0]` then fails to
          resolve, killing the whole graph before any CMake runs.
          The bind-mounted conan-cache from a previous run may
          already be in this state — that's why the command below
          re-exports all 7 recipes first (idempotent).

       b) `tee /tmp/cli-conf.log` writes inside the --rm container —
          gone the moment the build exits. Bind-mount a host dir
          (-v "$(pwd):/host") and write the log there.

       c) `bash -lc '...'` opens a login shell which prints
          `mesg: ttyname failed: Inappropriate ioctl for device`.
          Use `bash -c '...'`.

       Final form:
           mkdir -p "$(pwd)/conan-cache" "$(pwd)/output-arm"
           sudo docker run --rm \
               -v "$(pwd)/conan-cache:/root/.conan2" \
               -v "$(pwd)/output-arm:/work/conan-recipes/output" \
               -v "$(pwd):/host" \
               -e PROFILE=/work/conan-recipes/profiles/lin-gcc75-arm-linaro \
               -e PROFILE_BUILD=/work/conan-recipes/profiles/lin-gcc84-x86_64 \
               grpc-tc-mirror-arm bash -c '
                   for p in zlib abseil c-ares re2 protobuf openssl grpc; do
                       case "$p" in
                           zlib) v=1.3.1 ;; abseil) v=20250127.0 ;;
                           c-ares) v=1.34.6 ;; re2) v=20251105 ;;
                           protobuf) v=5.29.6 ;; openssl) v=3.4.5 ;;
                           grpc) v=1.78.1 ;;
                       esac
                       conan export /work/conan-recipes/$p --version=$v 2>&1 | tail -1;
                   done;
                   conan install --requires=grpc/1.78.1 \
                       -pr:h=$PROFILE -pr:b=$PROFILE_BUILD \
                       --build=missing --build="abseil*" --no-remote \
                       -s build_type=Release \
                       -o "*/*:shared=True" \
                       -c "*:tools.cmake.cmaketoolchain:user_toolchain=[\"/work/conan-recipes/profiles/toolchains/linaro-arm.cmake\"]" \
                       2>&1 | tee /host/cli-conf.log | tail -40;
                   echo "=== LINARO-ARM-TC count ===";
                   grep -c "==LINARO-ARM-TC==" /host/cli-conf.log
               '

       Interpreting the count:
       - count >= 3 (abseil + c-ares + zlib at minimum) → -c on CLI
         pushes the toolchain to the transitive abseil. Workaround:
         inject the same -c into run_test_grpc.sh Step 2 and Step 3.
       - count == 2 (still only c-ares + zlib) OR count == 0 with
         abseil clearly being rebuilt (new hash in the policy_checks
         error) → CLI -c does NOT help either. The bug is deeper
         than [conf] pattern parsing (Conan 2.27.1 strips
         user_toolchain from a transitive node under
         `--requires=grpc/...` regardless of where conf comes from).
         Move on to step 4.

    4. Last resort — build each package as a TOP-LEVEL requires.
       The consumer node always receives [conf]/`-c` correctly
       (that's what commit e9bdb19 proved with [8f]). So instead of
       one `--requires=grpc/...` that hides 6 nodes as transitive,
       build all 7 as separate consumer installs in dep-tree order.
       Each gets the toolchain → all binaries land in the cache →
       a final `--requires=grpc/...` then finds everything ready
       and does not rebuild anything.

           sudo docker run --rm \
               -v "$(pwd)/conan-cache:/root/.conan2" \
               -v "$(pwd)/output-arm:/work/conan-recipes/output" \
               -v "$(pwd):/host" \
               -e PROFILE=/work/conan-recipes/profiles/lin-gcc75-arm-linaro \
               -e PROFILE_BUILD=/work/conan-recipes/profiles/lin-gcc84-x86_64 \
               grpc-tc-mirror-arm bash -c '
                   for p in zlib abseil c-ares re2 protobuf openssl grpc; do
                       case "$p" in
                           zlib) v=1.3.1 ;; abseil) v=20250127.0 ;;
                           c-ares) v=1.34.6 ;; re2) v=20251105 ;;
                           protobuf) v=5.29.6 ;; openssl) v=3.4.5 ;;
                           grpc) v=1.78.1 ;;
                       esac
                       conan export /work/conan-recipes/$p --version=$v 2>&1 | tail -1;
                   done;
                   : > /host/cli-conf.log;
                   for ref in zlib/1.3.1 abseil/20250127.0 c-ares/1.34.6 \
                              re2/20251105 protobuf/5.29.6 openssl/3.4.5 \
                              grpc/1.78.1; do
                       echo "=== Building $ref as top-level ===" | tee -a /host/cli-conf.log;
                       conan install --requires="$ref" \
                           -pr:h=$PROFILE -pr:b=$PROFILE_BUILD \
                           --build=missing --no-remote \
                           -s build_type=Release \
                           -o "*/*:shared=True" \
                           2>&1 | tee -a /host/cli-conf.log | tail -5;
                   done;
                   echo "=== LINARO-ARM-TC count ===";
                   grep -c "==LINARO-ARM-TC==" /host/cli-conf.log
               '

       Expected: count == 7 (each top-level install runs CMake
       configure once, prints the marker once). If yes, follow up
       with the deployer install to produce .nupkg:

           sudo docker run --rm \
               -v "$(pwd)/conan-cache:/root/.conan2" \
               -v "$(pwd)/output-arm:/work/conan-recipes/output" \
               -e PROFILE=/work/conan-recipes/profiles/lin-gcc75-arm-linaro \
               -e PROFILE_BUILD=/work/conan-recipes/profiles/lin-gcc84-x86_64 \
               grpc-tc-mirror-arm bash -c '
                   conan install --requires=grpc/1.78.1 \
                       -pr:h=$PROFILE -pr:b=$PROFILE_BUILD \
                       --no-remote -o "*/*:shared=True" \
                       --deployer=/work/conan-recipes/extensions/deployers/legacy_nupkg.py \
                       --deployer-folder=/work/conan-recipes/output
               '
           ls -la output-arm/*.nupkg     # expect 7 files

       If the top-level loop builds all 7 successfully, the proper
       fix is to rewrite run_test_grpc.sh Step 2 to do exactly this
       loop instead of one --requires=grpc install — same output
       (.nupkg via deployer Step 3 unchanged).


[8i] Verify env-fallback patch (commits de4802b + 8278d55) actually
     reached the image AND landed in the generated toolchain:

    Symptom: after the env-fallback workaround for transitive
    user_toolchain (4b.1 + 4b.3 + arch gate) the run still shows
    `LINARO-ARM-TC count = 0` and the same `policy_checks.h "GCC 7
    or higher"` on a fresh abseil build folder. Two distinct causes
    look identical from the run log; this block tells them apart.

    1. Did the patch reach the built image?

           sudo docker run --rm grpc-tc-mirror-arm \
               grep -n "settings.arch" /work/conan-recipes/abseil/conanfile.py

       Expected (post-8278d55): one line containing
       `if _user_tc and str(self.settings.arch) in (` near
       `tc.blocks["user_toolchain"]`.

       - Empty output → image was cached with the OLD recipe layer.
         Rebuild with `--no-cache`:

             cd ~/conan-master
             sudo docker build --no-cache \
                 --build-arg BASE_IMAGE=$REGISTRY/library/gcc75-build-arm:0.1.0 \
                 -f Dockerfile.grpc-tc-mirror -t grpc-tc-mirror-arm .

         Then re-run the ARM build (NEXT_STEPS 4b.2). Stop here.
       - Line is present → patch is in the image. Move to check 2.

    2. Did the override actually land in conan_toolchain.cmake of
